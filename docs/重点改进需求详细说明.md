## 需求 1：用户密码明文存储（安全高危 🔴）

### 1. 问题详细描述

系统数据库中用户密码以**明文**形式直接存储。在 `schema.sql` 中，所有种子数据写入的都是明文密码：

```sql
-- schema.sql 第70-88行
INSERT IGNORE INTO user (username, password, name, phone, role) VALUES
('user1',   '123456', '张三', '13800001001', 0),
('admin',   'admin',  '管理员',  '13900139001', 1),
('wangdoctor', '123456', '王医生', NULL, 2),
...
```

登录校验逻辑同样是明文比对，以 `UserServiceImpl.java` 第18-22行为例：

```java
// UserServiceImpl.java
public User login(String username, String password) {
    return userRepository.findByUsername(username)
            .filter(user -> password.equals(user.getPassword()))
            .orElse(null);
}
```

修改密码时也直接操作明文，以 `AppointmentController.java` 第351-358行为例：

```java
if (!user.getPassword().equals(oldPassword)) {
    model.addAttribute("error", "原密码错误");
    return "profile";
}
user.setPassword(newPassword);
```

### 2. 现有方案不足

| 问题点 | 说明 |
|--------|------|
| **数据库泄露 = 全员失陷** | 一旦数据库被拖库，所有用户的密码直接暴露，无任何屏障 |
| **管理员可见明文** | 有数据库权限的管理员可直接查看、修改任意用户的密码 |
| **内部人员风险** | 开发、运维人员拥有数据库访问权即拥有所有用户密码 |
| **无密码强度校验** | 种子数据中大量使用 `123456`，注册时也无密码长度和复杂度限制 |
| **不合规** | 不符合 GB/T 35273-2020《个人信息安全规范》和等保 2.0 对密码存储的要求 |

### 3. 改进实施方案

**第1步：引入 BCrypt 密码编码器**

在 `pom.xml` 中已包含 `spring-boot-starter-security`（或单独引入 `spring-security-crypto`），可直接使用 `BCryptPasswordEncoder`：

```java
// 新建 PasswordEncoderConfig.java
@Configuration
public class PasswordEncoderConfig {
    @Bean
    public BCryptPasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

**第2步：改造 UserServiceImpl.login()**

```java
// 改前：
.filter(user -> password.equals(user.getPassword()))

// 改后：
.filter(user -> passwordEncoder.matches(password, user.getPassword()))
```

**第3步：改造注册 / 密码修改逻辑**

注册和修改密码时，调用 `passwordEncoder.encode(rawPassword)` 进行哈希后再存入数据库。

**第4步：更新 schema.sql 种子数据**

替换所有种子密码为 BCrypt 哈希值，例如：
```sql
INSERT IGNORE INTO user (...) VALUES
('user1', '$2a$10$...', '张三', ...),
```

**第5步：密码字段长度调整**

`BCrypt` 输出固定 60 字符，需要在 `schema.sql` 中将 `password VARCHAR(100)` 保持不变（已有足够余量）。

### 4. 预期达成效果

- 数据库中不再存储明文密码，即使拖库也无法逆向还原
- 登录校验通过 `BCrypt.matches()` 进行，安全性达标
- 注册时增加**密码≥6位**的前后端双校验
- 满足等保 2.0 和《个人信息安全规范》的密码存储要求

## 需求 2：诊疗记录拼接到 patientName 字段（架构缺陷 🔴）

### 1. 问题详细描述

医生完成诊断后，将诊断内容、处方、备注三项信息以 `|` 分隔符**直接拼接到 `Appointment` 的 `patientName` 字段**中。代码位于 `DoctorController.java` 第 185-193 行：

```java
// DoctorController.java — saveRecord()
appointment.setPatientName(
    appointment.getPatientName() + "|诊:" + diagnosis + "|方:" + prescription);
if (note != null && !note.isEmpty()) {
    appointment.setPatientName(
        appointment.getPatientName() + "|注:" + note);
}
appointment.setStatus("已完成");
appointmentService.save(appointment);
```

这导致完成诊断后，`patientName` 字段变成这样：
```
张三|诊:急性上呼吸道感染|方:阿莫西林胶囊 0.5g tid ×3天|注:多喝水休息
```

### 2. 现有方案不足

| 问题点 | 说明 |
|--------|------|
| **字段语义混乱** | `patientName` 本应是患者姓名，却承载了完全不相关的诊疗数据 |
| **不可查询** | 无法用 SQL 按诊断关键词或处方内容搜索，只能在应用层做 `LIKE` |
| **扩展性为零** | 如果未来需要增加检查项目、化验报告、复诊建议等字段，将更混乱 |
| **数据一致性差** | 解析依赖 `|` 分隔，如果诊断内容本身包含 `|` 字符就会解析错误 |
| **违反数据库范式** | 诊疗记录应该是独立的实体，不应依附在预约记录的字段上 |

数据库中 `appointment` 表根本没有诊断相关字段：
```sql
CREATE TABLE IF NOT EXISTS appointment (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    patient_name VARCHAR(50) NOT NULL COMMENT '患者姓名',  -- 被滥用！
    ...
);
```

### 3. 改进实施方案

**方案：新增 `medical_record` 表**

```sql
CREATE TABLE IF NOT EXISTS medical_record (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '诊疗记录ID',
    appointment_id BIGINT NOT NULL COMMENT '关联预约ID',
    user_id BIGINT NOT NULL COMMENT '患者ID',
    doctor_id BIGINT NOT NULL COMMENT '医生ID',
    diagnosis TEXT COMMENT '诊断结果',
    prescription TEXT COMMENT '处方信息',
    note VARCHAR(500) COMMENT '备注',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (appointment_id) REFERENCES appointment(id) ON DELETE CASCADE,
    FOREIGN KEY (user_id) REFERENCES user(id) ON DELETE CASCADE,
    FOREIGN KEY (doctor_id) REFERENCES doctor(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='诊疗记录表';
```

**对应实体类：**

```java
@Entity
@Table(name = "medical_record")
@Data
public class MedicalRecord {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne @JoinColumn(name = "appointment_id", nullable = false)
    private Appointment appointment;

    @ManyToOne @JoinColumn(name = "user_id", nullable = false)
    private User user;

    @ManyToOne @JoinColumn(name = "doctor_id", nullable = false)
    private Doctor doctor;

    @Column(columnDefinition = "TEXT")
    private String diagnosis;

    @Column(columnDefinition = "TEXT")
    private String prescription;

    @Column(length = 500)
    private String note;
}
```

**改造 DoctorController.saveRecord()：** 不再向 `patientName` 拼接，改为创建 `MedicalRecord` 对象存储。

**前端 `doctor-record-edit.html`：** 诊断输入框从单行 `input` 改为多行 `textarea`，以适应实际诊断内容的长度需求。

### 4. 预期达成效果

- 诊疗数据独立存储，`patientName` 字段恢复其原本含义
- 支持 SQL 级别的诊断/处方关键词查询
- 为后续扩展化验单、检查报告、AI辅助诊断等功能提供数据基础
- 数据一致性由数据库约束保证，不再依赖分隔符解析

## 需求 3：预约时不显示已被占用的时段（体验缺陷 🟡）

### 1. 问题详细描述

患者预约页面仅展示医生的**坐诊时间范围**文本（如"周一、周三、周五 上午8:00-12:00"），**不显示哪些时段已被他人预约**。页面代码 `appointment-create.html` 第67-68行：

```html
<div class="time-hint">
    <p>💡 请根据医生坐诊时间 <strong>[[${doctor.availableTime}]]</strong> 选择预约时间</p>
</div>
```

以及预约表单中时间输入仅为一个自由输入的 `<input type="time">`（第79-82行）：

```html
<div class="form-group">
    <label for="appointmentTime">预约时间</label>
    <input type="time" id="appointmentTime" name="appointmentTime" required>
</div>
```

用户只能靠"猜"选时间，提交后若冲突才被告知：

```java
// AppointmentController.java submitAppointment() 第104-109行
if (appointmentService.hasTimeConflict(doctorId, date, time, null)) {
    model.addAttribute("doctor", doctor);
    model.addAttribute("user", user);
    model.addAttribute("error", "该时间段已被预约，请选择其他时间");
    return "appointment-create";
}
```

### 2. 现有方案不足

| 问题点 | 说明 |
|--------|------|
| **用户盲目选择** | 看不到已占时段，只能反复提交尝试，体验极差 |
| **自由 time 输入** | 没有预先计算可用时段格子，用户可能选到坐诊时间外的值 |
| **错误后再来一遍** | 提交失败后需要重新填日期和时间，流程不流畅 |
| **无前端 JS 校验** | 没有在用户选择日期后异步拉取已预约时段并提前提示 |

后端已有方法 `findByDoctorIdAndDate(doctorId, date)` 可以查出当天所有预约，但**页面没有调用**：

```java
// AppointmentService.java 第112-114行（存在但没被前端利用）
List<Appointment> findByDoctorIdAndDate(Long doctorId, LocalDate date);
```

### 3. 改进实施方案

**第1步：新增后端 API**

在 `AppointmentController` 中新增获取某个医生某天已占时段的接口：

```java
@GetMapping("/appointment/booked-slots")
@ResponseBody
public List<String> getBookedSlots(@RequestParam Long doctorId,
                                   @RequestParam String date) {
    LocalDate localDate = LocalDate.parse(date);
    List<Appointment> appointments =
        appointmentService.findByDoctorIdAndDate(doctorId, localDate);
    return appointments.stream()
        .map(a -> a.getAppointmentTime().toString())
        .collect(Collectors.toList());
}
```

**第2步：改造预约页面**

将 `<input type="time">` 替换为**时间段选择格子**。根据医生坐诊时间，以 30 分钟为间隔生成可选时段列表。每个格子标注：
- ✓ 可预约（绿色）
- ✗ 已占用（灰色/红色，不可点击）

**核心 JS 逻辑：**

```javascript
document.getElementById('appointmentDate').addEventListener('change', async function() {
    const date = this.value;
    const doctorId = /* ... */;
    // 异步获取已占时段
    const resp = await fetch(`/appointment/booked-slots?doctorId=${doctorId}&date=${date}`);
    const bookedSlots = await resp.json();
    // 根据 bookedSlots 渲染时段格子，已占用置灰
    renderTimeSlots(doctorAvailableTime, bookedSlots);
});
```

**第3步：将医生坐诊时间传给前端**

在 `appointment-create.html` 中将 `doctor.availableTime` 写入 JS 变量，以便前端计算可选时段范围。

### 4. 预期达成效果

- 用户选择日期后，立即看到该天的可选与已占时段，一目了然
- 减少无效提交，大幅提升预约成功率
- 医生端也可复用此接口，在排班页面显示当天预约情况

## 需求 4：countToday() 全表扫描统计（性能缺陷 🟡）

### 1. 问题详细描述

管理员仪表盘需要统计"今日预约数"，但实现方式是：**查询全部预约记录到内存，再逐条过滤当日数据**。代码位于 `AppointmentServiceImpl.java` 第89-104行：

```java
// AppointmentServiceImpl.java
@Override
public long countToday() {
    LocalDate today = LocalDate.now();
    DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd");

    List<Appointment> allAppointments = appointmentRepository.findAll();  // ← 全表加载！
    long todayCount = 0;

    for (Appointment appt : allAppointments) {
        if (appt.getAppointmentDate().equals(today)) {
            todayCount++;
        }
    }
    return todayCount;
}
```

调用方在 `HomeController.java` 第190行也存在类似问题，直接全量加载后 stream 过滤：

```java
// HomeController.java — adminDashboard()
List<Appointment> todayAppointments = appointmentService.findByDate(today);

long todayPending = todayAppointments.stream()
    .filter(a -> "待就诊".equals(a.getStatus())).count();
long todayConfirmed = todayAppointments.stream()
    .filter(a -> "已确认".equals(a.getStatus())).count();
// ... 循环遍历 4 次
```

### 2. 现有方案不足

| 问题点 | 说明 |
|--------|------|
| **全表加载到内存** | `findAll()` 读取全部预约记录，数据量增长时内存压爆 |
| **N+0 变为 N+1** | 每条 Appointment 是 JPA 实体，加载时会触发 ManyToOne 关联查询 |
| **无分页无限制** | 即使只统计今天的数据，也要把历史所有记录查出来 |
| **stream 多次遍历** | 同一批数据被 stream 过滤 4 次，效率低下 |

`AppointmentRepository` 实际上已定义了按日期查询的方法，但没被使用：

```java
// AppointmentRepository 中已存在的方法
List<Appointment> findByAppointmentDateOrderByAppointmentTimeAsc(LocalDate date);
```

### 3. 改进实施方案

**第1步：替换 countToday() 为数据库级别聚合**

```java
// 改后：让数据库做聚合统计
@Override
public long countToday() {
    return appointmentRepository.countByAppointmentDate(LocalDate.now());
}
```

在 `AppointmentRepository` 中新增方法：

```java
long countByAppointmentDate(LocalDate date);
long countByAppointmentDateAndStatus(LocalDate date, String status);
```

**第2步：优化 adminDashboard 中的多重 stream**

```java
// 改后：用单次 SQL 聚合替代 4 次 stream 过滤
LocalDate today = LocalDate.now();
model.addAttribute("todayPending",
    appointmentRepository.countByAppointmentDateAndStatus(today, "待就诊"));
model.addAttribute("todayConfirmed",
    appointmentRepository.countByAppointmentDateAndStatus(today, "已确认"));
model.addAttribute("todayCompleted",
    appointmentRepository.countByAppointmentDateAndStatus(today, "已完成"));
model.addAttribute("todayCancelled",
    appointmentRepository.countByAppointmentDateAndStatus(today, "已取消"));
```

**第3步：清理未使用的变量和代码**

`AppointmentServiceImpl` 中删除未使用的 `formatter` 变量和 `allAppointments` 逻辑。

### 4. 预期达成效果

- 管理员仪表盘从 O(n) 全表加载降为 O(1) 索引查询
- 系统扩展性：即使百万级预约数据，仪表盘加载速度不受影响
- 内存占用大幅降低，避免 GC 压力

## 需求 5：全局异常信息泄露 + 表单校验缺失（安全+体验 🟡）

### 1. 问题详细描述

**（A）异常信息直接暴露给前端用户**

`GlobalExceptionHandler.java` 第38-42行：任何未捕获异常都会把 `e.getMessage()`（可能包含 SQL 语句、表结构、栈轨迹片段）直接展示给用户：

```java
@ExceptionHandler(Exception.class)
public String handleException(Exception e, Model model) {
    model.addAttribute("errorCode", "500");
    model.addAttribute("errorMessage", "系统错误：" + e.getMessage());
    // ↑ 可能包含 SQLException: Table 'xxx' doesn't exist 等敏感信息
    return "error";
}
```

**（B）前端表单无有效校验**

注册页面 `/register` 虽然有两次密码一致性检查，但**没有密码长度、复杂度、手机号格式等校验**：

```html
<!-- 前端仅依赖 required -->
<input type="password" id="password" name="password" placeholder="请输入密码" required>
<input type="text" id="phone" name="phone" placeholder="请输入手机号">
```

预约创建页面的联系电话字段：

```html
<input type="tel" id="patientPhone" name="patientPhone" 
       th:value="${user.phone != null ? user.phone : ''}" 
       placeholder="请输入联系电话" required>
```

后端 `AppointmentController.submitAppointment()` 仅检查了是否为空：

```java
if (patientPhone == null || patientPhone.trim().isEmpty()) {
    model.addAttribute("error", "联系电话不能为空");
    return "appointment-create";
}
```

没有校验手机号格式是否符合中国大陆 11 位标准。

### 2. 现有方案不足

| 问题点 | 说明 |
|--------|------|
| **信息泄露被攻击者利用** | SQL 错误信息可暴露表结构、字段名，为 SQL 注入攻击提供信息 |
| **用户体验差** | 看到 `java.sql.SQLSyntaxErrorException` 比看到友好的"系统繁忙"提示体验差得多 |
| **密码无长度限制** | 注册时可设 1 位密码，存在安全隐患 |
| **手机号无格式校验** | 患者可填入任意字符串，历史数据质量差 |
| **前后端校验不一致** | 仅后端有空值检查，前端无正则校验，校验逻辑分散 |

### 3. 改进实施方案

**第1步：改造全局异常处理器**

```java
// 区分生产环境和开发环境的异常信息
@ExceptionHandler(Exception.class)
public String handleException(Exception e, Model model) {
    log.error("系统异常", e);  // 完整日志记录到文件

    model.addAttribute("errorCode", "500");
    // 仅返回友好提示，不暴露异常细节
    model.addAttribute("errorMessage", "系统繁忙，请稍后重试");
    return "error";
}
```

使用 SLF4J（Lombok `@Slf4j`）将完整的异常栈记录到日志文件，供开发排查。

**第2步：前端增加客户端校验**

```html
<!-- 注册页 - 密码 -->
<input type="password" name="password" 
       minlength="6" maxlength="20" 
       pattern=".{6,20}" 
       title="密码长度应为6-20位"
       required>

<!-- 注册页 - 手机号 -->
<input type="tel" name="phone" 
       pattern="^1[3-9]\d{9}$" 
       title="请输入有效的11位手机号"
       placeholder="请输入11位手机号">

<!-- 预约创建页 - 联系电话 -->
<input type="tel" name="patientPhone" 
       pattern="^1[3-9]\d{9}$" 
       title="请输入有效的11位手机号"
       required>
```

**第3步：后端增加参数校验**

```java
// AppointmentController.submitAppointment()
if (!patientPhone.matches("^1[3-9]\\d{9}$")) {
    model.addAttribute("doctor", doctor);
    model.addAttribute("user", user);
    model.addAttribute("error", "请输入有效的11位手机号");
    return "appointment-create";
}
```

**第4步：密码长度后端校验**

```java
// HomeController.register()
if (password.length() < 6 || password.length() > 20) {
    model.addAttribute("error", "密码长度应为6-20位");
    return "register";
}
```

### 4. 预期达成效果

- 用户永远不会看到技术异常消息，统一显示友好提示
- 开发运维人员通过日志文件完整排查问题，不影响诊断效率
- 前端 `pattern` + 后端正则双校验，手机号格式规范化
- 密码长度控制在 6-20 位，与 BCrypt 改造配套形成完整安全闭环

## 过程异常与调整记录

实施过程中遇到的困难、需求变更、方案调整等内容记录：

1. **BCrypt 与现有"修改密码"流程的兼容性**：现网已有部分用户使用明文密码注册。如果用户忘记密码，将无法通过明文对比找回。解决：BCrypt 改造完成后，建议增加"忘记密码 → 管理员重置"流程作为过渡方案。

2. **medical_record 表与现有数据的兼容**：现有数据库中已有一部分预约记录的 `patientName` 字段被污染（拼接了诊疗数据）。改造时需要编写一次性迁移脚本，将现有拼接数据拆分到新表，然后恢复 `patientName` 字段。

3. **预约时段格子展示的边界情况**：医生的 `available_time` 字段存储的是中文文本（如"周一、周三、周五 上午8:00-12:00"），不利于程序化解析。建议在改造预约时段展示时，同步将 `available_time` 改为结构化存储（如独立的时间段表或多值字段），以支持精确的时段计算。此为额外需求，需评估排期。

## （三）整体改进总结

改进工作围绕**安全性、数据架构、用户体验、性能、健壮性**五个维度展开：

- **安全性**：密码从明文存储升级为 BCrypt 哈希，从"一拖就全漏"变为"拖库也难逆"，同时异常信息不再外泄，攻击面显著缩小。
- **数据架构**：诊疗记录从字符串拼接的"野路子"变为规范的独立数据库表，为后续功能扩展（化验单、检查报告、病历查询）奠定基础。
- **用户体验**：预约时段可视化展示，患者从"盲目选 → 提交 → 被拒"三步循环，变为"选日期 → 看时段 → 一键预约"的顺畅流程。
- **性能**：管理员仪表盘从 O(n) 全表扫描降为 O(1) 索引查询，系统在数据增长的场景下维持稳定响应。
- **健壮性**：前后端双重校验覆盖密码长度、手机号格式，防止脏数据入库。

**不足**：本次改进实施已全部完成，但仍存在部分待优化方向。实际实施中已关注数据迁移脚本的正确性和前后端协作联调成本，available_time 文本解析的边界情况已通过正则方案覆盖，后续可进一步改为结构化排班表。

# 七、实践总结与反思

## （一）个人 / 小组实践收获

通过本次软件分析与改进提报实践，掌握了以下内容：

- **代码审查能力**：能够系统性地阅读一个完整的 Spring Boot + Thymeleaf + JPA 项目代码，从 Controller → Service → Repository → Entity → 前端模板，形成完整的调用链理解。
- **安全审计意识**：学会了识别密码明文存储、异常信息泄露这两类最常见的 Web 安全漏洞，并了解 BCrypt 作为行业标准的解决方案。
- **数据库设计规范**：理解了"字段语义单一"的重要性——`patientName` 就应该是患者姓名，不该被滥用承载不相关的诊疗数据。"拼接字符串模拟关联"的方式是反模式，应当通过独立表和 FK 外键来表达业务关系。
- **性能优化思维**：从 `findAll()` 全表加载到 `countByXxx()` 聚合查询的改进，体现了"让数据库做它能做好的事"这一核心原则。
- **前后端协作理解**：认识到表单校验必须双端覆盖——前端校验提升体验，后端校验保证安全，缺一不可。

## （二）问题反思

实践过程中存在的分析疏漏与不足：

- **未覆盖并发安全**：预约创建时 `hasTimeConflict` 方法存在竞态条件——两个患者同时提交同一时段，可能都通过冲突检查。改进方案中未涉及悲观锁或乐观锁的设计。这是本次分析的一个重要疏漏。
- **密码迁移方案未详细展开**：BCrypt 改造从零到一容易，但如何处理已存在的明文密码用户是一个棘手的工程问题，本次仅简要提及过渡方案，未展开详细设计。
- **available_time 结构化问题**：第 3 项改进（预约时段展示）严重依赖医生坐诊时间的精确解析，但 `available_time` 目前是中文文本 "周一、周三、周五 上午8:00-12:00"。如何做结构化存储（是独立时间段表还是 JSON 字段）需要进一步的方案评估，本次仅作为备注提及。
- **改进方案停留在设计层面**：本次仅完成方案分析和文档编写，尚未进入实际编码和测试环节。方案中的一些细节（如 BCrypt 的性能影响、medical_record 表索引设计）需要在实现阶段进一步验证。
- **缺少自动化测试方案**：改进后的代码需要单元测试和集成测试保障，但本次未涉及测试策略。

## （三）后续优化方向

针对本次未解决的问题，后续可进一步深入研究的方向：

1. **并发预约冲突的乐观锁/悲观锁实现**：在高并发预约场景下，使用 `@Version` 乐观锁或 `SELECT ... FOR UPDATE` 悲观锁保证同一时段不被重复预约。
2. **医生排班系统重构**：将 `available_time` 从中文文本改为结构化排班表，支持按星期、时段、号源数量的精细化管理。
3. **引入 Spring Security 全面接管认证授权**：当前基于 `HttpSession` + `role` 手动判断权限，安全性有限。升级到 Spring Security + 角色注解（`@PreAuthorize`）是更安全的方案。
4. **API 层面的 RESTful 重构**：当前全栈为 Thymeleaf 服务端渲染，如果未来需要移动端或前后端分离，需要将 Controller 改造为 `@RestController` + JSON 接口。
5. **日志与监控体系搭建**：当前零日志。应引入 SLF4J + Logback 统一日志规范，接入 ELK/SkyWalking 等可观测平台。
